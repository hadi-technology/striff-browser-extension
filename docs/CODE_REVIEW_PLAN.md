# Striffs Extension — Maintainability & Reliability Review + Remediation Plan

**Date:** 2026-07-09 · **Branch:** re-work · **Version:** 1.0.3

## Scope & relationship to the existing review

There is already a `striffs-production-readiness-review (1).md` in the repo. That doc is **bug- and launch-blocker-focused** (wrong URLs, data-loss in comment submit, client-side preview contradicting the design, etc.). **This document is deliberately a different lens**: dead code, simplification, code organization, documentation, systematic consistency, GitHub-UI future-proofing, logging, and user-facing error UX. Where the two overlap it is noted so we don't fix the same thing twice. Do the launch-blockers from that doc first; this plan is the reliability/maintainability layer under it.

Grades: **P0** = fix before public release, **P1** = this release cycle, **P2** = follow-up/polish.

## 1. Code organization

### 1.1 `src/striffs.js` is a 10,834-line hand-edited monolith with *fake* module boundaries — **P1**
The file contains markers like `// ---- src/striffs-core.js ----`, `// ---- src/striffs-state.js ----`, `// ---- src/striffs-comment-mode.js ----`, and an inline `// ---------- Comment panel (inlined from striffs-comment-panel.js) ----------`. **None of those source files exist**, and there is **no concat/build step** (`scripts/package-extension.mjs` only zips; no bundler in `package.json`). The markers imply a modular build that was never wired up, so every edit happens in one 10.8k-line file that ships verbatim as a content script.

Impact: hard to review, high merge-conflict surface, the memory note "auto-generated by build script" is not actually true, and reviewers/new contributors are misled by the section headers.

**Plan (pick one, don't leave it as-is):**
- **Option A (recommended): make the modularity real.** Split at the existing marker boundaries into `src/striffs/*.js`, add a tiny concat step (esbuild `--bundle` or a 20-line concat script) invoked by `npm run build` and by `package-extension.mjs`, and load the built file. Each module already assigns onto the shared `window.Striffs` (`S`) namespace, so ordering is the only constraint — encode it in the build.
- **Option B (cheap): drop the pretense.** Remove the fake-file markers, replace with honest `// ===== SECTION: Comment mode =====` banners, and update the memory note. Keeps one file but stops lying about structure.

### 1.2 `popup.js` and `options.js` are near-duplicate token UIs — **P2**
`html/shared.js` already factors out `validateToken`/`saveToken`/`proxyJson`, but both pages still carry parallel `setStatus`, badge-refresh, save-handler, and clear-handler code with only string differences (`"No token set"` vs `"No token saved"`, `"GitHub error"` vs `"GitHub /user error"`). Move the status/badge/handler wiring into `shared.js` and parameterize the strings.

## 3. Logging — **P1**

- **`cwarn`/`cerr` are NOT gated by the debug flag** (`striffs.js:928-929`), while `clog`/`cinfo`/`debugDump` are. Net effect: a normal user's console gets `[Striffs]` warnings/errors in production. Decide the policy: either (a) gate `cwarn` behind the debug flag too and reserve `cerr` for genuinely actionable failures, or (b) keep warn/error always-on but audit every call site so nothing benign (e.g. `readCachedUpdatedAt failed`, `updateFileTreeAvailability failed`) logs on the happy/expected path.
- **`buildPathIdMapping` alone has 14 verbose `clog` dumps** (`[buildPathIdMapping] ...`, `[map] ...`) printing SVG node counts and full map arrays. Even gated behind debug this is a lot of noise for one function; consolidate into a single structured `debugDump("pathIdMapping", {...})`.
- **Two logging conventions coexist**: `S.cwarn?.(...)` in most of the file vs a destructured `const { cwarn } = S;` in the `striffs-pr` / `striffs-dom-ui` sections. Pick one so grep-ability and behavior are uniform.
- **Standardize the prefix.** Some calls already pass `'[Striffs]'` as an argument *and* the wrapper prepends `'[Striffs]'`, yielding `[Striffs] [Striffs] ...` (e.g. `striffs.js:832`, `850`, `1375`). Remove the redundant literal at call sites.

## 4. GitHub-UI future-proofing — **P1**

The good news: DOM coupling is already centralized in a frozen `S.SELECTORS` object with **ordered fallback arrays** per concept (toolbar, filesNode, filesRoot, diffContainers, fileLinks, fileTreeItems). That is the right pattern and should be preserved. The risks:

- **Reliance on hashed CSS-module classes** (`section[class*="PullRequestFilesToolbar-module"]`, `div[class*="Diff-module__diff"]`, `prc-Stack-Stack-UQ9k6`, `prc-Textarea-TextArea-snlco`, `prc-StateLabel`). The prefix-match approach survives hash-suffix churn, but a module *rename* breaks it silently with no signal.
- **No telemetry when all fallbacks miss.** When every selector in a group fails, the extension just quietly doesn't mount. There's no counter or breadcrumb to tell you GitHub shipped a breaking DOM change.

**Plan:**
- Add a single `S.resolveSelector(group)` helper that walks the fallback array and, on total miss, emits **one** throttled `cwarn` + increments an engagement/breadcrumb counter (e.g. `selectorMiss:toolbar`). This turns a silent breakage into an observable signal — the difference between "we noticed GitHub's redesign in a day" and "users filed 1-star reviews for a week."
- Add a short comment block at the top of `SELECTORS` documenting the **layering contract** (new-UI selectors first, legacy fallbacks last) so future edits keep the order meaningful.
- Consider a lightweight CI/manual "selector smoke" check (the existing `test:ui`/`test:visual` live tests partly cover this) that fails loudly if the primary selector for each group stops matching on a known PR.

## 5. Systematic-consistency issues (reliability) — **P1**

These are the "no single approach to one problem" items. Several are also flagged in the existing production-readiness review (M-1 through M-4); calling them out here because they're the highest-leverage reliability work:

- **Caching is split across three stores** (chrome.storage, IndexedDB, in-memory) with **reads picking the freshest** and **four different cache-clearing implementations** (`clearChromeStorageCaches`, `clearIndexedDbCaches`, `clearRuntimeCaches`, plus the in-content clear path), **two IndexedDB DB names**, and most "Clear Cache" paths not actually clearing IndexedDB. Consolidate to **one cache module with one `clearAll()`** that every entry point calls. This is the single biggest correctness-under-time risk in the codebase.
- **Service-worker module-scope state** in `background.js` assumes a long-lived worker; MV3 kills it. Audit each module-level variable (debug flag, token cache, temp-key tracking) for "must survive a worker restart" and move those to storage. `loadDebugFlag()` being fire-and-forget is one instance.
- **`striffsTempChangedFiles` keys can leak** if the worker dies mid-flight; `cleanupOrphanedTempKeys` exists but confirm every write path is covered.

## 6. User-facing error messages — **P1**

The `describeApiError` taxonomy (`striffs.js:9301+`) is genuinely good — status/errorCode-driven, with distinct tooltip/toast/tone/disabled per case and a token-aware branch. Keep it. Gaps:

- **Raw backend text is passed straight to users** in several branches (`FORBIDDEN`, `INVALID_ARGUMENT`, `SOURCE_PARSE_FAILED`, `INTERNAL_ERROR`, `502`, and the fallthrough all do `tooltip: text, toast: text`). `extractHumanMessage` only unwraps JSON — it doesn't guarantee the string is user-appropriate. Define a small allow-list of "safe to show verbatim" codes; for everything else show a friendly generic message and log the raw text via `debugDump`.
- **The 50 MB cap message is misleading** (existing review M-5) — it says the PR is too large when the real limit is a client-side size cap. Fix the wording.
- **Native `confirm()`** is used for at least one important prompt (existing review M-11); replace with the extension's own toast/inline UI so it's not blocked/inconsistent across browsers.
- **Consistency pass:** make sure every terminal error both (a) sets a button tooltip via `reconcileStriffButtonState` and (b) shows a toast, with matching tone. Right now `disabled`/`tone` differ case-by-case in ways that look accidental (e.g. `SOURCE_PARSE_FAILED` stays enabled, `INVALID_ARGUMENT` disables) — document the intended rule.

## 7. Documentation — **P2**

- **Zero JSDoc in `striffs.js`** (0 `/**` blocks across ~704 function definitions). Not asking for full coverage, but the ~15 load-bearing entry points (`bootIfNeeded`, `autoFetchStriffs`, `buildPathIdMapping`, `maybePrefetchStriffs`, the cache module, `describeApiError`, the comment-mode submit flow) each deserve a 3-line contract comment: what it assumes about DOM state, what it mutates on `S`, and its failure mode.
- **README "Runtime Layout" section describes the fake module split** — update it once §1.1 is resolved so the docs match reality.
- **Update the project memory note** that claims `striffs.js` is "auto-generated by build script" (it isn't).

1. **§2 hygiene** + **§3 double-prefix logging fix** — trivial, immediate, visible. (P0/P1)
